一、严重（阻塞发布）

严重 C1 构建 缺少代码签名配置
tauri.conf.json 中无 signingIdentity（macOS）/ certificateThumbprint（Windows）配置。产物未签名 → macOS Gatekeeper 拦截，Windows SmartScreen 警告，无法通过公证。
macOS 配置 Developer ID + notarize 环境变量；Windows 配置证书指纹 + sha256 + timestampUrl。
严重 C2 构建 dfu-util 未配置 sidecar 打包
bundle 中无 externalBin 字段。核心功能"固件刷写"依赖 dfu-util，但安装包不含此工具 → 普通用户无法完成 DFU 刷写。
将各平台 dfu-util 二进制放入 src-tauri/bin/，配置 "externalBin": ["bin/dfu-util"]，代码中用 sidecar 调用。
严重 C3 构建 资源包未配置打包且内容残缺
bundle 中无 resources 字段。games-pack 仅含 README（无 .fap），themes-pack 目录为空。打包后"资源导入"功能将无法定位源文件。
声明 "resources": ["resources/*"]；补齐真实 .fap/.asset 文件；Rust 侧用 resource_dir() 解析路径。
严重 C4 构建 缺少 release 编译优化配置
Cargo.toml 无 [profile.release] 段。未启用 LTO / strip / panic=abort，debug 二进制已达 44MB，release 产物预计 30-50MB+，含调试符号。
添加 opt-level="z" + lto=true + codegen-units=1 + strip="symbols" + panic="abort"。

二、高危

高危 H1 安全 API Key 明文存储（无加密无钥匙串）
ai_config.json 以明文 JSON 存储 API Key，文件权限默认 0644，任何同用户进程可直接读取。无 Keychain / Credential Manager 集成。
使用 keyring crate 存入系统钥匙串；或文件权限设 0600 + AES-GCM 加密（密钥派生自机器特征）。
高危 H2 安全 SSRF：用户可控 API URL + Key 泄露
get_api_url() 中自定义 URL 无任何校验（不校验协议、不校验内网地址）。Bearer {api_key} 随 Authorization 头 POST 到该 URL → 可被诱导发往攻击者服务器或探测内网（169.254.169.254 云元数据）。
强制 HTTPS；维护 provider 域名白名单；禁止私网/回环/链路本地地址；自定义 URL 需二次确认。
高危 H3 安全 AI 脱敏器 7 种模式存在多处绕过 + 图片未脱敏
① 裸 hex（UID/NFC Key）不触发 ② DMS 坐标不触发 ③ ghp_/AKIA 等 token 不触发 ④ 国际手机号不触发 ⑤ 全角/HTML实体/Base64 编码绕过 ⑥ ai_chat_with_image 的 image_base64 原样发送，截图中的密钥/密码无法检测 ⑦ 仅脱敏 user 消息，AI 回复回显不脱敏。
增加无前缀裸 hex 启发式检测；增加 DMS/ghp_/AKIA 格式；图片发送前 UI 弹窗确认；AI 回复也做轻量扫描。
高危 H4 代码 cancel_import 完全失效（取消导入是空操作）
cancel_import 仅设置 AtomicBool 标志，但 run_import_pipeline 和 run_virtual_import 中无任何一处读取该标志。6 个阶段、storage_write 分块循环均无取消检查点。用户点"取消"无效果。
在 pipeline 每个阶段入口 + storage_write 循环内检查 cancel_import_flag.load()，为 true 时 bail 并清理半成品文件。
高危 H5 代码 cancel_flash 无法中止刷写 + dfu-util 子进程泄漏
cancel_flash_flag 仅在 progress_cb 回调里检查（且只 return 回调不中止流程）。run_dfu_util（数分钟）、wait_for_device_reconnect（最长 60s+）、storage_write 均无检查点；dfu-util 子进程无 kill 机制 → 取消后进程继续运行至完成。
flasher 的 stderr 读取循环 + reconnect 轮询中检查 cancel；cancel 时 child.kill()；用 tokio::select! + CancellationToken。

三、中等

中等M1安全 CSP 策略过于宽松
script-src 'unsafe-inline' 允许内联脚本；img-src https://* 允许任意 HTTPS 图片（可数据外泄）；connect-src http://localhost:* 开发配置泄漏到生产；Google Fonts 与 CSP 不一致。
移除 unsafe-inline；img-src 收窄为 'self' data: blob:；移除 http://localhost；字体本地化。
中等M2安全 save_log_dump 任意文件写入
file_path 直接来自前端 IPC 无校验，std::fs::write 可写入任意路径（如 ~/.ssh/authorized_keys）。
校验后缀（仅 .txt/.log）；限制写入目录；拒绝 .. 路径。
中等M3安全 tar 包路径穿越 → 设备端文件覆盖
entry_path_str 直接拼接到设备路径，恶意 tar 含 ../ 条目可穿越到设备系统目录。
用 Path::components() 拒绝 ParentDir；用 path-clean 清理；拼接后校验前缀。
中等M4安全 多模态图片未脱敏直接发送
ai_chat_with_image 仅脱敏文字消息，image_base64 原样发送。截图中的密钥/密码（像素级）无法检测。
发送图片前 UI 弹窗确认"截图将原样发送"；可选集成本地 OCR 做脱敏检查。
中等M5安全 shell:allow-open 权限过宽
允许前端通过 shell.open 打开任意 URL/文件，无 scheme 白名单。
配置 scheme 白名单（仅 https:）；审计所有 shell.open 调用点。
中等M6代码 flash_firmware 阻塞 tokio 运行时
async fn 内同步调用 reqwest::blocking / thread::sleep / Command::output()，阻塞 worker 线程数分钟，饿死 AI 流式、事件 emit、屏幕镜像。
用 tokio::task::spawn_blocking 包裹同步刷写流程，进度通过 channel 转发。
中等M7代码 dfu-util stdout 管道未读取 → 潜在死锁
子进程 stdout 被 Stdio::piped() 但从不读取，超过 64KB 管道缓冲时 child.wait() 永久挂起。
并发 drain stdout，或直接 Stdio::null()。
中等M8代码 screen_mirror 可重复启动 → 竞态
无"已在运行"检查，重复调用 start 产生两个并发镜像任务：虚拟模式帧翻倍，真实模式串口锁竞争。
start 时检查 *running，已为 true 则直接返回；用 CancellationToken 替代 Mutex<bool>。
中等M9代码 临时文件未清理 + 并发覆盖
dfu 解压文件从不删除；tar 文件在 stage 4/5 报错时残留；固定文件名 {package_id}.tar 并发导入互相覆盖。
用 tempfile::NamedTempFile 或 uuid 命名；用 scopeguard 在函数退出（含错误路径）统一删除。
中等M10代码 device connect/disconnect 未停旧会话
直接覆盖 rpc_session，未发 StopSession RPC → 设备侧 session 残留，快速重连可能 seq 错乱。
替换/断开前 best-effort 调用 stop_session()（忽略错误）。
中等M11代码 config 非原子写入 → 损坏风险
std::fs::write 直接覆盖，写入中崩溃/并发写入会截断 config.json，无文件锁。
写临时文件后 fs::rename 原子替换；用 Mutex 序列化。
中等M12代码 RPC 端口失效时忙等 + 镜像错误被静默吞掉
rpc_send_recv_raw 在硬断开端口上形成忙等自旋直到 deadline；next_frame 把错误吞为 Ok(None)，镜像变黑但无错误反馈。
区分可恢复/不可恢复错误，后者立即返回 Err 上抛；不可恢复时 emit screen-mirror-error 并停止。
中等M13构建 生产 JS bundle 包含约 600 行 Mock 代码
mockInvoke 及其依赖的 MOCK_* 数据全部打包进生产 JS，增大 bundle + 暴露数据结构。虽不会执行（isTauri() 为 true），但 tree-shaking 无法移除。
用 import.meta.env.DEV 守卫，或拆到独立 dev-only 入口，Vite 会整段剔除。
中等M14构建 第三方依赖版本重复
reqwest 同时存在 0.12 + 0.13；windows crate 同时存在 0.57/0.58/0.61 三个版本。增大体积 + 扩大攻击面。
统一 Cargo.toml 版本声明；排查传递依赖来源。
中等M15构建 依赖漏洞未自动核验
未运行 cargo audit / npm audit。已核查版本较新但未通过 advisory-db 验证。
CI 中加入 cargo audit + npm audit --omit=dev；发布前修复所有 high/critical。

四、低风险（建议改进）

低L1代码lib.rs 中 3 处 unwrap 有 panic 风险
573/721 行有 is_empty() 守卫（安全）；651 行 path.file_name().unwrap() 有 is_file() 守卫（基本安全）。但 panic=abort 模式下任何 panic 直接终止进程。
改用 .cloned() / .and_then() / .unwrap_or_default()。
低L2安全dfu-util PATH 劫持风险
PATH 查找到的 dfu-util 无签名/哈希校验，恶意同名程序可被执行。
优先用随包内置版本；打印完整路径供用户确认；考虑 Rust 原生 DFU 库。
低L3安全unzip 外部命令依赖系统安装
zip 解压依赖系统 unzip，Linux/Windows 不一定有。跨平台一致性差。
引入 zip crate 做纯 Rust 解压。
低L4安全Google Fonts 与 CSP 不一致
index.html 加载 Google Fonts CDN，但 CSP 未包含对应域名 → 要么字体被阻断（功能缺陷），要么 CSP 未生效（安全缺陷）。
字体文件本地化，移除 CDN 引用。
低L5安全仅脱敏 user 消息，AI 回复回显风险
AI 在回复中回显用户 UID/密钥等数据时不会被脱敏，存入历史后再次发送给云端。
对 assistant 消息也做轻量扫描（至少匹配 UID/API Key/NFC Key 格式）。
低L6代码大量死代码（维护性地雷）
后端 RpcCommandId 枚举标注 dead_code 但运行时用裸整数；前端 5 个 store 的 initListeners + useAutoConnect 均未调用。若误启用 useAutoConnect 会与模块级自动注册叠加 → 每个事件触发两次。
删除未用代码，统一为单一注册路径。
低L7代码mirrorStore 模块级常驻定时器
1s 间隔定时器模块加载时即启动，无论镜像是否激活都永久运行。
改为 startMirror 时启动、stopMirror 时停止。
低L8代码chatStore 流式监听器泄漏
快速连续发送两条消息时，第二次 .then 覆盖第一次的 unlisten → 监听器泄漏 + 后端继续消耗 token。
进入前先 streamUnlisten?.()；AiChat 卸载时也清理。
低L9代码sanitizer 的 7 处 Regex::new().unwrap()
硬编码字面量当前不会 panic，但模式笔误时首次调用即 panic（无错误恢复）。
已用 OnceLock 懒加载（安全）；可改 expect("正则编译失败") 更明确。
低L10代码driver.rs 为 mock 桩实现
三个 Windows 驱动函数返回硬编码提示文本，未真正调用 zadig。driver.rs:12 有 TODO。
接入真实 zadig 流程，或标注为"手动安装指引"并移除 TODO。
低L11代码deviceStore.demoConnect 的 setTimeout 未清理
未保存句柄、未在卸载/HMR 时清理。
保存 timer 句柄并在适当时 clearTimeout。
低L12代码firmwareStore 跨 store 订阅未取消
模块级 useDeviceStore.subscribe(...) 永不取消，随 app 生命周期常驻。
在回调内做状态过滤，避免无意义触发。
低L13构建bundle 元数据不完整
缺 copyright；无 macOS minimumSystemVersion / Windows webviewInstallMode。
补充 copyright；按需设置平台细分配置。
低L14构建Vite 缺少高级优化
无 manualChunks / chunkSizeWarningLimit / cssMinify。
添加 manualChunks 拆分 vendor；设 chunkSizeWarningLimit: 500。
低L15构建缺少自定义 macOS Info.plist
无自定义 Info.plist。Tauri 2 构建时自动生成。串口/CDC 设备访问不需要 USB entitlement（非阻塞）。
可补充自定义 Info.plist 写入版权信息、版本号等元数据。

六、DMG/EXE 加密方案可行性分析

结论：方案部分可行，但无法完全防止逆向

Rust 后端天然保护良好；前端资源是主要薄弱点；"加密"不是标准做法，应组合多种手段。

各层保护能力

层级当前状态逆向难度加固建议
Rust 后端二进制 已编译为原生机器码 ★★★★★ 高 启用 strip + LTO 即可，核心逻辑已难以逆向
前端 JS/HTML/CSS 明文嵌入二进制 ★☆☆☆☆ 极低 可用 javascript-obfuscator 混淆；Tauri 2 支持自定义 asset 协议可做运行时解密

AI Prompt / 路由逻辑 前端 stores 含部分逻辑 ★★☆☆☆ 低 将 prompt 模板移到 Rust 后端；前端仅做 UI 展示
IPC 协议定义 前后端共享类型 ★★★☆☆ 中 无法隐藏（Tauri 架构决定），但影响有限

推荐保护方案（分层组合）

#措施效果成本
①[profile.release] 启用 strip + LTO后端二进制体积减半 + 符号剥离0

关键限制

① 没有绝对不可破解的桌面应用 — 只要有足够时间和动力，任何本地运行的程序都可被逆向
② Tauri 架构决定了前端资源必须以某种形式存在于二进制中 — 即使加密，运行时也需解密到内存，可被 dump
③ 真正的护城河应是后端服务 — 将核心逻辑（AI prompt 工程、资源生成）放到云端 API，客户端仅做 UI 和设备通信
④ Flipper Zero 社区本身是开源生态 — .fap/.asset 文件格式公开，加密这些文件的 ROI 很低
⑤ DMG/EXE 本身的"加密" — 通常指磁盘镜像加密（macOS）或 PE 加壳（Windows），对 Tauri 应用不适用，Tauri 产物是标准安装包

建议策略

先完成 ①②③（零/低成本），再评估 ④⑤⑥ 的 ROI。当前阶段优先修复审核发现的 39 个问题，再进入打包流程。核心 IP 保护应走云端化路线而非本地加密。
